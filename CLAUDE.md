# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
# Build
dotnet build

# Run all tests
dotnet test

# Run a single test class
dotnet test tests/NSchema.Core.Tests --filter "FullyQualifiedName~DefaultSchemaComparerTests"

# Run a single test method
dotnet test tests/NSchema.Core.Tests --filter "FullyQualifiedName~DefaultSchemaComparerTests.Compare_AddsTable"
```

Targets `net10.0`. Warnings are errors in the test project. The solution (`NSchema.Core.slnx`) contains only `src/NSchema.Core` and `tests/NSchema.Core.Tests`. The Postgres provider was extracted to its own repository (commit `dd3e695`); only Core lives here.

### Tests

xUnit v3 with **Shouldly** (assertions), **NSubstitute** (mocks), **Verify.XunitV3** (snapshots), and **Testcontainers.PostgreSQL** (real-database integration tests under `tests/.../EndToEnd`, which spin up a Postgres container even though the provider lives elsewhere). `using` directives for `Xunit`/`Shouldly`/`NSubstitute`/`VerifyXunit` are global.

**Snapshot tests are the convention for every output surface** (diff renderer, plan renderer, DDL writer, state serialization, …). They live next to the test as `*.verified.txt` in a `Snapshots/` folder. When intended output changes, update the snapshot (Verify writes a `*.received.txt` on mismatch; accept it to update the `*.verified.txt`).

## Architecture

NSchema is a declarative database schema migration library for .NET (the engine behind the NSchema CLI). The user describes the schema they want in **SQL DDL**; NSchema introspects the database, diffs, and applies the difference.

`NSchemaApplication.CreateBuilder()` returns an `NSchemaApplicationBuilder` (its surface is split across the `NSchemaApplicationBuilder.*.cs` partials: `.Schemas`, `.State`, `.Policies`, `.Configuration`), which uses a `HostApplicationBuilder` internally purely to compose configuration, logging, metrics, and DI. `Build()` produces an `NSchemaApplication` (a plain `IDisposable`, **not** an `IHost`). It does not run as a host: the app is a thin facade exposing `Services`, `Operations` (`INSchemaOperations`), `Locks` (`IStateLockManager`), `CurrentSchema` (`ICurrentSchemaProvider`), `Project` (`IProjectProvider`), `PlanFile` (`IPlanFileManager`), and `State` (`ISchemaStateManager`). Every operation is reached through `app.Operations` — each method takes an `{Name}Arguments` record and returns `Task<Result<{Name}Result>>`, carrying success/failure and diagnostics as data rather than throwing. There is no `BackgroundService` and no host lifecycle. The app instance is reusable — it may be invoked multiple times across its lifetime.

### Layer separation

The codebase splits into a **domain layer** (pure planning) and an **application layer** (orchestrating a run):

- **Domain layer**, one namespace per pipeline stage:
  - `Project/` — the schema model (`Project/Domain/Models/`, rooted at `DatabaseSchema`), the DDL reader/writer/parser (`Project/Ddl/`), the project provider (`ProjectProvider` — the project is the declared desired state), the current-schema provider, and `ISchemaPolicy`.
  - `Diff/` — the structured diff model (`Diff/Domain/Models/`, rooted at `DatabaseDiff` — **the complete current→desired difference**: the schema changes plus, on its root `Scripts` list, the script runs those differences imply; matched nodes reference their script by name via `MigrationScript`, resolved with `DatabaseDiff.FindScript` — names are validated unique project-wide, so the name is the key), `ISchemaComparer` (the pure structural compare), `IProjectComparer` (internal; composes the structural compare with run-once resolution and change-event matching — `CurrentState`, the schema plus the recorded `ScriptExecution`s, is its input and lives here), `IDiffPolicy` (`Diff/Policies/`), and the diff reader (`DiffReader`, which projects a `DatabaseDiff` into a renderer-neutral `DiffDocument`, scripts included).
  - `Plan/` — the plan model (`Plan/Domain/Models/`, rooted at `MigrationPlan` — **the single complete artifact**: the complete diff and the SQL statements that execute it), `IPlanLinearizer`, the saved-plan-file machinery (`Plan/PlanFile/`), and `IMigrationPlanner` (default `MigrationPlanner`) returning `Result<MigrationPlan>` (policy diagnostics ride on the `Result`). `MigrationAction`s are pipeline-internal currency between the linearizer and the dialect — public as the dialect's vocabulary, but not part of the artifact. The planner knows nothing about operations or run orchestration.
  - `Sql/` — `ISqlDialect` (renders one action to statements; provided by a provider package and **required for planning**) and `ISqlExecutor`. Core ships no dialect.
- **Application layer:**
  - `Operations/` — one vertical slice per operation (see below).
  - `IMigrationWorkflow` (`Operations/Services/`) — the imperative shell operations share.
  - `Configuration/` — the generic config-in-SQL model (`ConfigBlock`/`ConfigValue`); Core only carries capability, the CLI interprets it. `BACKEND` and `PROVIDER` are the only configuration statements — the top-level statement set is closed (no catch-all config capture; an unknown keyword — including any retired spelling — is a plain syntax error, with no tombstone cases in the grammar).

`DiffReader` is a **public, stateless utility, not a DI service** — Core never consumes it; it exists for consumers (the CLI's presenter is its only user). It's a plain `new`-able class with a shared `.Default` singleton, so a caller reads a diff without touching the container. `Read(DatabaseDiff)` returns a `DiffDocument` — an ordered list of `DiffLine`s, each carrying its `ChangeKind` (null for a blank spacer), nesting `Depth`, and already-formatted `Text`, plus the aggregate `DiffSummary`. The document is **renderer-neutral**: a front-end folds it into colour, Markdown, or plain text by mapping each line's kind, never re-parsing rendered text. There is no `IDiffReader` interface and no `Use*` builder method. Core ships **no** schema or SQL-plan text renderer — those `SchemaRenderer`/`SqlPlanRenderer` dumps were trivial presentation with no reuse value and moved to the CLI; a consumer wanting to render those models (or a different diff format) writes its own.

### Operations

Each operation is its own vertical slice: an **internal** `IOperation<TArgs, TResult>` handler (`Execute(args, ct)`), a **public** `{Name}Arguments` record (the discoverable home for its inputs, empty where it has none yet), and a **public** `{Name}Result` record. Handlers are registered as `IOperation<{Name}Arguments, Result<{Name}Result>>` and invoked through the public `INSchemaOperations` facade (`app.Operations`) — not by user code resolving the handler. Operations return `Result`/`Result<T>` and never throw for expected outcomes, print, lock, or prompt: they narrate transient progress through `IProgress<OperationProgress>` and leave rendering, locking, and confirmation to the caller. There is no shared non-generic `IOperation` marker and no `OperationKind` enum — adding an operation means a new slice folder (handler + arguments + result), a `TryAddSingleton<IOperation<…>, …>()` registration, and a method on `INSchemaOperations` / `NSchemaOperations`.

Built-in operations (`src/NSchema.Core/Operations/`):

- **`PlanOperation`** (`Plan`) — computes a plan without executing. `PlanArguments.Target` (`PlanTarget`) selects what: `Recorded` (the default — a preview against recorded state; empty at bootstrap, so a first plan previews a full create); `Live` (against the live database, required — what `Apply` plans with); `Teardown` (a teardown of the managed schema — always the recorded state; empty state means nothing is managed and the teardown is empty — unscoped, the `terraform plan -destroy` analogue, bypassing diff/plan policies). All plan targets require a configured state store (the diff is computed against the schema plus the run-once ledger). `OutFile` also writes a **saved plan file** (forward or teardown; a blocked plan is still written — apply enforces the policies again, so the file is a review artifact, not a bypass).
- **`ApplyOperation`** (`Apply`) — requires a configured state store (state records the run-once ledger, which is not rebuildable; a disposable database opts out explicitly via `UseEphemeralState()`), re-runs the diff policies against the plan's carried diff (policy errors block unless `ApplyArguments.Force`, which demotes them to warnings), then executes `ApplyArguments.Plan`'s statements via `ISqlExecutor` and captures state; the plan comes from a plan operation or a saved plan file (applied verbatim, no recompute — the policy check is the only recomputation). A saved teardown flows through this same path, so **tearing down is `Plan(Teardown)` → `Apply`** — there is no separate destroy operation, and since a teardown diff is fully destructive, applying one requires `Force` under the default destructive-action policy.
- **`RefreshOperation`** — captures the live schema to the state store without planning or applying.
- **`ValidateOperation`** — loads the desired schema and validates it against registered `ISchemaPolicy` implementations; no planning/applying.
- **`DriftOperation`** — reads recorded (offline) state and live (online) schema and compares (direction recorded → live, so an out-of-band add reads as `Add`), returning the `DatabaseDiff`. Requires both a state store and a live provider. A pure observation — no policies run. The analogue of `terraform plan -refresh-only`.
- **`ImportOperation`** — fetches the live schema (optionally filtered by `Schemas`) and writes it under `OutputDirectory` as SQL DDL via `DdlWriter`, **one file per major object** (e.g. `app/tables/users.sql`, `app/routines/add_tax.sql`), with schema-level objects in a per-schema header (`app.sql`) and database-global extensions in top-level `extensions.sql`. Additive: absent objects are left in place, re-imported objects merge (incoming wins). The file-split/merge logic lives directly in the handler — no `ISchemaImportTarget` seam.
- **`DoctorOperation`** — runs read-only health checks against the configured infrastructure and reports the outcome of each.

There is no `Show`, `Destroy`, `PlanDestroy`, or `ForceUnlock` operation. Previewing and applying a teardown are `Plan(Teardown)` / `Apply` above; reading recorded state or a saved plan, inspecting or force-releasing the lock, and state surgery (pull/push, taint/untaint) are thin front-end reads and read-mutate-write loops over the public seams — `app.CurrentSchema`, `app.PlanFile`, `app.Locks`, and `app.State` — not Core operations. Confirmation is likewise a caller concern (the CLI prompts); Core operations neither prompt nor lock.

**`IMigrationWorkflow`** (`Operations/Services/`) is the imperative shell around the pure planner shared by `Plan`, `Apply`, `Refresh`, and `Validate`: `ComputePlan(currentSource, schemas)` and `ComputeTeardown()` resolve the desired and current schemas and run `IMigrationPlanner` (returning `Result<MigrationPlan>`), `Validate()` runs just the schema policies, and `Refresh()` captures the live schema to the store, returning `Result<StateCapture>` (a failure when no store is configured). `Drift`, `Import`, and `Doctor` do **not** use the workflow — they talk to `ICurrentSchemaProvider` / `ISchemaComparer` / the importer directly. The workflow does not lock, confirm, or render.

### Schema providers

**The project** — the declared desired state — comes exclusively from SQL DDL files ("desired" needs no qualifier: the project *is* what you want). `AddProjectSource(baseDirectory, glob|Matcher)` registers a `DdlSchemaSource`; the **public** `IProjectProvider` (`app.Project`, default `ProjectProvider`) globs every registered source, reads matched files with `DdlReader`, and aggregates into a `ProjectDefinition` (public) — the declared `DatabaseSchema` **plus** the scripts declared inline. **The `SCRIPT` statement** (`SCRIPT '<name>' RUN [ALWAYS | ONCE] ON <event> AS $$…$$;`) is the only script form, and there is **one script model**: `Script` (name + SQL + the `ScriptEvent` it runs on). `ON PRE|POST DEPLOYMENT` parses to a `DeploymentEvent` (with `DeploymentPhase`), `ON ADD COLUMN|ALTER COLUMN TYPE|ADD CONSTRAINT <path>` to a `ChangeEvent` (with `ChangeTrigger` + schema/table/member path — the "data migration" case); the `RUN` clause parses to `RunCondition` (`UNLESS EXISTS` is reserved and rejected). **`RUN ONCE` is ledger-enforced, and diffing is symmetric:** the differ compares what-I-have to what-I-want, where `CurrentState` (internal, `Diff/`) is the current schema **plus** the recorded executions — so the diff carries schema changes *and* script runs in one artifact. An already-executed run-once declaration is not part of the current→desired difference, so the planner drops it (a changed body warns and stays skipped — rename to re-run) and a skipped migration neither splices, decomposes, nor suppresses hazards. `DatabaseDiff.Scripts` lists every script the difference implies, in declaration order (matched-only for change-event scripts by construction — an unmatched one is not part of the difference), and matched diff nodes carry the script's name. The state envelope carries the ledger as `scripts` (envelope stays version 1; payloads without it — including pre-5.0 `executedScripts` payloads — read as an empty ledger). `ScriptExecution` (name + body hash + when) is **shared script vocabulary** (`Project/Domain/Models/Scripts/`) — an execution record is identity, not declaration (the body may since have changed or gone) — used by the ledger and the differ alike, so the recorded executions pass straight through to planning with no boundary translation. **The `MigrationPlan` is the complete executable artifact** — the complete diff (scripts included) and the statements travel as one, so everything that moves the plan (plan result, plan file, apply arguments) moves all of it; there is no separate manifest and no parallel run-once parameters anywhere. A successful apply passes the applied plan to `IMigrationWorkflow.Refresh(applied, force, ct)`, which derives the run-once ledger records (`Diff.Scripts.Where(RunCondition == Once)` → name + hash) at the state boundary; a failed or empty apply passes nothing and records nothing. Refresh never rebuilds the ledger, and an unreadable existing payload **fails the capture unless forced** (`RefreshArguments.Force`; apply's post-execution capture always forces — the SQL has already run) — a forced replacement resets the ledger and rides the successful capture `Result` as warning diagnostics (the read errors demoted to warnings — error severity would flip the result to a failure), which the refresh/apply operations forward onto their own results. Planning requires a state store (the ledger is half of the current state). `Script` exposes a public computed `Hash` — the canonical body hash (SHA-256 via the internal `ScriptHashing`) that a run-once execution is recorded under; it is the **only** hash surface. The pre-4.4 spellings (`PRE|POST DEPLOYMENT '<name>' AS $$…$$;`, `MIGRATION ['name'] FOR <trigger> <path> AS $$…$$;`) were **removed in 5.0**: each now fails the parse with an error naming the `SCRIPT` replacement, and script names are required (anonymous migrations died with the old grammar). `GetProject` returns `Result<ProjectDefinition>` — the pure planner receives only the `ProjectDefinition` payload, and `MigrationWorkflow` merges the read diagnostics into plan/validate outcomes (nothing currently produces read diagnostics; the seam stays for parse diagnostics). Script names are validated unique project-wide after template expansion. `GetProject(scope, ct)` is its one method — a public *read* of the resolved project (the CLI's `script untaint` takes declaration hashes from it); *supplying* desired schema from code remains deliberately unsupported — DDL files are the only input, and there is no `ISchemaTransformer`. Multiple `AddProjectSource` calls aggregate (e.g. a base set plus an environment overlay).

The DDL stack lives in `Project/Ddl/`: `DdlLexer` → `DdlParser` (partials per statement family: `.Create`, `.Drop`, `.Grant`, `.Config`, `.Scripts`, `.Templates`, `.SchemaAccumulator`) → schema model. **Schema templates** (`TEMPLATE name BEGIN … END;` + `APPLY TEMPLATE name IN SCHEMA a, b;`) declare a reusable object group instantiated per schema: bodies are fully parsed (no views/schemas/extensions/drops inside; unqualified names bind to the target schema, qualified ones escape), and the internal `TemplateApplicator` expands applications in `ProjectProvider` after all sources aggregate — location- and order-agnostic, invisible to diff/plan/import. `SCRIPT`s of either event kind are allowed in template bodies — change-event scripts use an unqualified `table.member` path, and the table need not be declared in the same template (another template applied to the same schemas may declare it — matching is decided at diff time); deployment-bookend scripts instantiate one `Script` per applied schema, scoped to it via `DeploymentEvent.AppliedSchema` (a hand-written deployment script is global), so a scoped read filters every script by its event alone. The `{schema}` token substitutes in both the SQL and the script name (names must end up unique per instance). **Table templates** (`TEMPLATE name FOR TABLE BEGIN <members> END;`) hold comma-separated table members instead of statements and merge into a table via an `INCLUDE name` member (columns land at the include position); includes resolve after schema-template application, so a templated table can itself include one. Duplicate index-backed names per schema (a hazard when a mixin carries an index) are caught by `StructuralIntegritySchemaPolicy`, templates or not. `DdlWriter` renders the model back to DDL (used by Import). `DdlFormatter` is a gentle token-stream reformatter (backs the CLI `fmt` command) — it pretty-prints without going through the full parse/model round-trip, so it preserves what it doesn't understand.

**Current-state** access goes through `ICurrentSchemaProvider`. The live source implements the public `ISchemaIntrospector`, used **only** for current-state:

```csharp
ValueTask<DatabaseSchema> GetSchema(SchemaSourceMode preferred, string[]? schemaNames, bool required = true, CancellationToken cancellationToken = default);
```

- `Online` — reads the live database provider (registered via `UseCurrentSchema<T>()`). Throws if unconfigured.
- `Offline` — reads the state-backed source (requires a registered `ISchemaStateStore`). Throws if unconfigured.
- `Offline, required: false` — prefers offline, falls back to online, throws only if neither exists.

The internal `DefaultCurrentSchemaProvider` wires both. Registering a state store via `UseStateStore*()` is all that's needed to enable offline planning. `UseEphemeralState()` registers an in-memory store (and matching lock) for disposable databases — integration tests, container bootstraps — which satisfies apply's state-store requirement without persisting anything beyond the application instance.

`ISchemaStateStore` deals only in **serialized payloads** (`Read`/`Write` of `ReadOnlyMemory<byte>`) — the backend SPI, a persistence sink (file, blob, …) that never sees the schema model. Core owns the format behind the consumer-facing seam, **`ISchemaStateManager`** (`app.State`, in `NSchema.Current.Storage` alongside its arguments/result records — the storage capability's consumer surface), which composes the store with the internal `ISchemaStateSerializer` and deals in the **public `SchemaState` model** (`Current/Domain/Models/` and `Current/Locks/` — the captured schema plus the run-once execution ledger, with `RecordScripts`/`RemoveScript`/`FindScript` mutators). Its typed `Read`/`Write` are the primary surface — front-ends and library consumers do state surgery as pull → mutate → push (taint = remove a ledger entry, untaint = record one); `ReadRaw`/`WriteRaw` move the payload uninterpreted for byte-shuffling (a raw push validates the payload deserializes, then writes it **verbatim** — never a re-serializing round-trip). Every member takes an `{X}Arguments` record and returns `Result<{X}Result>`: no store configured and an unreadable payload are failures, not exceptions — except `ReadRaw`, which never interprets, so corrupt state can still be pulled for repair. The seam does not lock (caller-managed, as with operations). `MigrationWorkflow` consumes the manager (a plan with an unreadable ledger fails rather than re-planning recorded scripts; a *forced* refresh is the corrupt-state recovery path — see the scripts passage above), while `CurrentSchemaProvider`'s offline read and `DoctorOperation` still use store + serializer directly. `SchemaStateEnvelope` (the versioned wire format) stays internal.

### State locking

`IStateLock` (`Current/Locks/Backends/IStateLock.cs`) is the **backend seam** coordinating exclusive access to shared state. It is separate from `ISchemaStateStore`, but one backend may implement both. `Acquire(StateLockRequest, ct)` returns an `IStateLockHandle` and throws `StateLockedException` (carrying the holder's `StateLockInfo` when readable) when the lock is already held; `Release(ct)` drops whatever lock is held; `Peek(ct)` reads the holder without acquiring. The handle exposes `Info` and an idempotent `Release(ct)` — it is **not** `IAsyncDisposable`, so a caller can acquire a lock and never release it to hold it past the current process (a long-lived manual lock).

Front-ends never touch `IStateLock` directly; they go through `IStateLockManager` (`app.Locks`), which wraps it: `Acquire(request, skipLock)` returns a `Result<IStateLockHandle>` (a failure carrying the holder's details when already locked; a no-op handle when there is no backend, or when `skipLock` is set — the latter carrying a warning naming the lock it ran past), and `Peek` / `Release` (force-release) mirror the backend. **Locking is caller-managed** — Core operations never lock; the CLI acquires the lock through `app.Locks` around the state-mutating runs it guards (apply, refresh, and destroy-via-apply) and releases it in a `finally`.

**The lock is registered automatically alongside the store.** `UseFileStateStore(path)` also registers a matching `FileStateLock` at `<path>.lock`; `UseStateStore<T>()` / `UseStateStore(instance)` register the same instance as the lock when the backend also implements `IStateLock`. An **explicit** lock choice (`UseStateLock*` / `UseFileStateLock`, tracked via a private `_explicitStateLock` flag) is never overridden by store registration, regardless of call order. With no store and no explicit lock, **no `IStateLock` is registered** and the manager's `Acquire` yields a no-op handle (there is no `NoOpStateLock`). (`FileStateLock` is a local-dev lock-file, not a distributed lock.)

### Planner pipeline

`MigrationPlanner` (`Plan/Domain/MigrationPlanner.cs`) conducts the pipeline, producing `Result<MigrationPlan>` — the complete artifact — with any `PolicyDiagnostic`s riding on the `Result`. **A policy block is a failure that still carries the complete plan** ("may not apply", not "stopped computing"): every stage runs regardless, and error severity is what blocks application — the CLI decides how to act. **The line between the stages: Diff answers "what is different" — schema changes and the script runs they imply; Plan answers "how it executes" — order and SQL.** Stages:

1. **Schema stage** — runs every `ISchemaPolicy` against the desired schema. Also exposed standalone as `IMigrationPlanner.Validate(desired)` (used by the validate operation).
2. **Diff stage** — `IProjectComparer` produces the complete `DatabaseDiff`: `ISchemaComparer` emits the structural tree directly (no flat-action intermediate), run-once resolution drops already-executed declarations (their skip warnings and dead-migration infos are diff-stage diagnostics), and the matcher attaches pending change-event scripts to their nodes by name, with every implied run on the diff's root `Scripts` list. Every `IDiffPolicy` then validates the complete diff — a matched backfill legitimately suppresses the data-hazard warning because script coverage is diff information. The built-in `DestructiveActionDiffPolicy` runs here and enforces `DestructiveActionOptions.Policy`.
3. **Linearize stage** — `IPlanLinearizer` (default `PlanLinearizer`) walks the diff and emits `MigrationAction`s in a safe dependency order, weaving the diff's own scripts in as `ExecuteScript` actions: matched change-event scripts splice at their name-referenced changes, deployment scripts bookend the list (pre first, post last).
4. **Render stage** — the planner walks the actions in order and renders each through the required `ISqlDialect` (one action in, one or more statements out — decomposition, never reordering), scripts included: an `ExecuteScript` typically renders as its verbatim `Statement` (the action carries that default), but a dialect may validate or normalize the SQL. The artifact is the diff plus the statements; the actions themselves are discarded.

There are **no transformer seams**: the desired schema, diff, and plan are each validated by policies but never rewritten by the pipeline.

## When writing documentation and XML/code comments.

- Public facing changes should be included in CHANGELOG.md.
- Don't overdocument everything. A brief, single-line summary is enough for XML docs. In some advanced cases where explanation is actually needed, then add another sentence or two in a Remarks block.
- When writing code comments, be aware that repeating the same reasoning in multiple places creates coupling and very quickly becomes out of date, so keep comments restricted to what actually matters at the place they're written.
- Don't expose implementation details in documentation or changelogs, as documentation is a form of contract, so changing an implementation becomes a breaking change.
- Stop confirming negatives. If I ask you to change something like "you've accidentally coupled these two services", don't then go and add five comments in various places saying "these two classes aren't coupled", because nobody was expecting them to be in the first place.
