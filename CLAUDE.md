# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
# Build
dotnet build

# Run all tests
dotnet test

# Run a single test class
dotnet test tests/NSchema.Core.Tests --filter "FullyQualifiedName~DefaultDatabaseComparerTests"

# Run a single test method
dotnet test tests/NSchema.Core.Tests --filter "FullyQualifiedName~DefaultDatabaseComparerTests.Compare_AddsTable"
```

Targets `net10.0`. Warnings are errors in the test project. The solution (`NSchema.Core.slnx`) contains only `src/NSchema.Core` and `tests/NSchema.Core.Tests`. The Postgres provider was extracted to its own repository (commit `dd3e695`); only Core lives here.

### Tests

xUnit v3 with **Shouldly** (assertions), **NSubstitute** (mocks), **Verify.XunitV3** (snapshots), and **Testcontainers.PostgreSQL** (real-database integration tests under `tests/.../EndToEnd`, which spin up a Postgres container even though the provider lives elsewhere). `using` directives for `Xunit`/`Shouldly`/`NSubstitute`/`VerifyXunit` are global.

**Snapshot tests are the convention for every output surface** (diff renderer, plan renderer, DDL writer, state serialization, …). They live next to the test as `*.verified.txt` in a `Snapshots/` folder. When intended output changes, update the snapshot (Verify writes a `*.received.txt` on mismatch; accept it to update the `*.verified.txt`).

## Architecture

NSchema is a declarative database schema migration library for .NET (the engine behind the NSchema CLI). The user describes the schema they want in **SQL DDL**; NSchema introspects the database, diffs, and applies the difference.

`NSchemaApplication.CreateBuilder()` returns an `NSchemaApplicationBuilder` (its surface is split across the `NSchemaApplicationBuilder.*.cs` partials: `.Schemas`, `.State`, `.Policies`, `.Configuration`), which uses a `HostApplicationBuilder` internally purely to compose configuration, logging, metrics, and DI. `Build()` produces an `NSchemaApplication` (a plain `IDisposable`, **not** an `IHost`). It does not run as a host: the app is a thin facade exposing `Services`, `Operations` (`INSchemaOperations`), `Locks` (`IStateLockManager`), `Database` (`IDatabaseProvider`), `Project` (`IProjectProvider`), `PlanFile` (`IPlanFileManager`), and `State` (`IDatabaseStateManager`). Every operation is reached through `app.Operations` — each method takes an `{Name}Arguments` record and returns `Task<Result<{Name}Result>>`, carrying success/failure and diagnostics as data rather than throwing. There is no `BackgroundService` and no host lifecycle. The app instance is reusable — it may be invoked multiple times across its lifetime.

### Layer separation

The codebase splits into a **domain layer** (pure planning) and an **application layer** (orchestrating a run):

- **Domain layer**, one namespace per pipeline stage:
  - `Project/` — the schema model (`Project/Domain/Models/`, rooted at `Database`; every object name is an `SqlIdentifier` — a value object whose identity is the exact written text (ordinal, case-sensitive equality; `users` and `Users` are different names), with deliberately no implicit string conversions, so records containing names compare exactly and no call site chooses a comparison — the planner warns when a declared name matches an observed one only up to case (the misspelled-adoption near-miss); in source, quoting (`"Order Details"`, `""` escaping a quote) carries the characters a bare identifier cannot and escapes the vocabulary, but never changes identity — the writer quotes only what needs it (non-identifier-shaped text and the member-opening keywords); every piece of opaque SQL the model carries verbatim (bodies, definitions, expressions, predicates) is a `SqlText` — same value-object rules, ordinal equality (it's data); both derive from the abstract `ValueObject` record (one `Value`, equality by value, renders as its value — each derived type owns its comparison semantics); naming follows one rule — a **name** (`SqlIdentifier`) is a bare label scoped to its container, a **reference** is a name as written at a use site (possibly unqualified; `RoutineReference`), and an **address** is absolute and identifies from outside the tree: `ObjectAddress` (schema-level object), `MemberAddress` (table member, `Owner` giving the containing object's address), and `ScopedAddress` (a named item in a schema or global at the database root — scripts) under the abstract `Address`, which is kind-free because which kinds may share a location is engine-specific — `ObjectKind` carries the *what* beside the address wherever full identity is needed — `ObjectIdentity(Kind, Schema, Name)` is that pair flattened, with a computed kind-free `Address`; there is deliberately no `ObjectKind.Unknown` (an identity always knows its kind; a location that cannot know one is an `ObjectAddress`, and set membership stays exact equality, never wildcard matching); on the tree itself, the nodes are **mutable classes forming a parent-linked graph** under an inheritance hierarchy (structure by inheritance; marker interfaces reserved for capabilities): abstract `DatabaseElement` is name+comment plus the one enforced copy primitive — `abstract Clone()`, covariantly overridden per type as the whole deep-copy surface (no `With*`/`Rename` variants; reshaping is property assignment) — **the model has no constructors**: nodes are built with object initializers, `required` members mark what a node cannot exist without (`Name`, `Column.Type`, `View.Body`, …), and owned-collection properties lazily create attached instances so a default-constructed node's collections still wire parents — `DatabaseObject : DatabaseElement` (the seven kinds + `Extension`) adds abstract `Kind`, a `Schema` parent reference, and the single shared computed `Identity`, and `DatabaseMember : DatabaseElement` (columns, constraints, indexes, triggers) adds the owning object as `Parent`; child node lists are **owned collections** (`DatabaseObjectCollection<T>`/`DatabaseMemberCollection<T>`, built on `Collection<T>`'s virtual hooks) that adopt on add and detach on remove/clear, while every other list (identifier lists, grants, enum values, fields, index keys — and `Database.Schemas`/`Extensions`, which wire nothing) is a plain get-only `List<T>` — collection properties are never replaced after construction, only mutated; scalars are set — attaching a node (or a collection) that already belongs elsewhere **throws** (at most one parent at a time; a move is an explicit remove-then-add, and nothing is copied silently); the **ownership rule** governs mutation: only mutate a tree you built or cloned — a tree handed to you (the observation, the project) is shared, so the rewrite passes (aligner, import merge, the linearizer's backfill variant) clone first, while the assembly pipeline mutates its own aggregate in place; equality is hand-written per type — structural over the declared definition, excluding parent and comment — with the standing rule that nodes never key hash sets or dictionaries; parent references never serialize (`[JsonIgnore]` on the two parent edges — a statement of the model, since parents are derivable from position and rewired on read; the serializers' only modifier is the principled computed-property rule) and Verify snapshots skip them (a node's snapshot location is its parentage). Value objects/addresses/identities/scripts/directives/diffs/state stay records), the DDL reader/writer/parser (`Project/Ddl/`), the project provider (`ProjectProvider` — the project is the declared desired state), the current-schema provider, and `IProjectPolicy` (`Project/Policies/`). **The language separates declarations from directives**: statements declare state (CREATE/GRANT/SCRIPT), directives steer management — the renames: `RENAME <kind> <current> TO <name>;` (kinds SCHEMA/TABLE/COLUMN/VIEW/ENUM/DOMAIN/TYPE/SEQUENCE/FUNCTION|PROCEDURE|ROUTINE→routine; current side fully qualified, target always bare — renames never move across containers). **There are no `DROP` or `PARTIAL` statements** (removed in 5.0, hard delete — they fail as unknown statements): removal is absence from the project, gated by the managed identity set in state, and adoption is per-object (declare an existing object and apply). Directives never enter the schema tree (`Database`/`Schema` are pure observation vocabulary — no `OldName`); they assemble onto `ProjectDefinition.Directives` (`ProjectDirectives`, structured by the **level** a directive acts at, not per kind: `SchemaRenames` (container level), flat kind-tagged `Renames` (`ObjectRenameDirective`, each carrying an `ObjectKind` — the public enum naming the seven schema-level kinds), `ColumnRenames` (member level), and the scripts; kind-generic consumers loop `ObjectKind` once instead of copy-pasting per-kind blocks). Directive addresses name **current reality** — what exists now. `DirectiveValidator` enforces the static rules at the read (target declared, source not still declared, no chains/duplicate sources/duplicate targets/self-renames, no directives into undeclared schemas — all findings at once, resolved through schema renames); the differ reports an applied rename (source gone, target present) as a self-expiry info, like a dead migration block. `DatabaseLookup` (`NSchema.Model.Services`, internal) is the address→node lookup service (`Has(ObjectKind, ObjectAddress)` plus the typed `Find*` reads).
  - `Diff/` — the structured diff model (`Diff/Domain/Models/`, rooted at `DatabaseDiff` — **the complete current→desired difference**: the schema changes, with each change-event script inlined on the node it accompanies (`MigrationScript`, a whole `ChangeScript`), and the deployment (bookend) scripts on the root `DeploymentScripts` list (`DeploymentScript`s); `ChangeScripts()` walks the tree to re-collect the inlined ones — there is no central script list and no name lookup), `IDatabaseComparer` (the structural compare of two observations — `Compare(current, desired)`, where the current side is an `AlignedDatabase`: the observation rewritten into the declared name-space plus the `RenameLog` of what moved, so matching is exact-name only and the log stamps `RenamedFrom` — the comparer knows nothing about directives or management; drift and teardown pass `AlignedDatabase.Unaligned`), the internal `DatabaseAligner` (applies the rename directives up front, producing the aligned tree + log; an ambiguous rename — source still declared, or target name taken — is *skipped* with an error diagnostic rather than thrown, and an already-applied rename (source gone, target present) self-expires with an info), the internal `ChangeScriptDecorator` (attaches each change-event script to the modified-table member diff it accompanies, as a pass over the computed diff), `IProjectComparer` (internal; orchestrates Align → Compare → Decorate with run-once resolution — `CurrentState`, the full observation plus the recorded `ScriptExecution`s and the managed `IdentitySet`, is its input and lives here), and the diff reader (`DiffReader`, which projects a `DatabaseDiff` into a renderer-neutral `DiffDocument`, scripts included).
  - `Plan/` — the plan model (`Plan/Domain/Models/`, rooted at `MigrationPlan` — **the single complete artifact**: the complete diff and the SQL statements that execute it), `IPlanLinearizer`, the saved-plan-file machinery (`Plan/PlanFile/`), and `IMigrationPlanner` (default `MigrationPlanner`) returning `Result<MigrationPlan>` (policy diagnostics ride on the `Result`). `MigrationAction`s are pipeline-internal currency between the linearizer and the dialect — public as the dialect's vocabulary, but not part of the artifact. The planner knows nothing about operations or run orchestration.
  - `Sql/` — `SqlDialect` (the abstract per-action rendering base a provider subclasses — one overridable method per migration action, named after it; **required for planning**) and `ISqlExecutor`. Core ships no dialect, but the base renders the ANSI-standard actions itself through an overridable quoting kernel; abstract methods mark the SQL every dialect must supply, and the rest return `Unsupported` (a blocking error) until overridden — or `Skipped` (warn and emit nothing), the dialect's choice.
- **Application layer:**
  - `Operations/` — one vertical slice per operation (see below).
  - `IMigrationWorkflow` (`Operations/Services/`) — the imperative shell operations share.
  - **Configuration is its own grammar.** `PLUGIN`, `ENGINE`, `DATABASE`, and `STATE` parse only under the config grammar (`NsqlReader.ReadConfig`/`ReadConfigFile` → `Result<NsqlConfigDocument, NsqlDiagnostic>` — the app-free typed config read, usable before an application exists) and are rejected in project files with an error naming the split. A plugin receives `PluginConfig` (`NSchema.Plugins` — label + typed attributes, translated from the parsed statement by the config assembly); the generic `ConfigBlock` is gone. The top-level statement set of each grammar is closed — an unknown keyword is a plain syntax error. **`PLUGIN` separates dependency declaration from configuration**: `PLUGIN <label> ( source = '<package id>', version = '<range>' );` declares the package (label required — it is the name `DATABASE`/`STATE` reference; both attributes required, the version in interval notation as NuGet writes it — no floating versions — with bare-version-means-exact; ranges are the `VersionRange` value object (structural equality — `[5.0,6.0)` equals `[5.0.0, 6.0.0)` — canonical `ToString` feeding resolution, `Satisfies(SemanticVersion)` for evaluation) over the hand-rolled public `SemanticVersion` (equality agrees with SemVer precedence), no NuGet package dependency), and `ENGINE ( version = '<range>' );` (no label, at most one) asserts the *host's* version — evaluated by the host via `EngineConfig.Requirement.Validate(SemanticVersion)`, never implicitly by Core, so library mode is unaffected. The config side mirrors the project layering — parser → reader (pure syntax) → assembler (validation + domain model): `ConfigAssembler.Assemble(documents)` validates every rule (statement shape, labels/packages declared once, at most one `ENGINE`/`DATABASE`/`STATE`, every `DATABASE`/`STATE` label resolves to a `PLUGIN` or a Core built-in — the `file` state store) and returns `Result<ConfigDefinition, NsqlDiagnostic>` — plugins, engine requirement, database/state settings, best-effort on failure. It runs host-side over every config document resolved together, because merge scope is host knowledge; `ReadConfig` itself validates nothing. `ConfigProvider.GetConfig(paths)` is the one-call entry point (read every file → assemble, all failures reported at once, best-effort value) — a host that layers configuration (env overlays) instead reads, merges, and calls `Assemble` itself. **Configuration is its own slice** (`Config/`, `NSchema.Config`): it owns the configuration domain — the definition, its assembly, its diagnostics, and everything minted from config statements (`PluginDeclaration`, `EngineConfig`/`EngineRequirement`, the `VersionRange` and `SemanticVersion` value objects) — so engine-level configuration (transaction mode, policy/convention config) lands there without touching the plugin SPI. The domain models are pure — only the assembler touches the syntax tree, so `Plugins/` has no Nsql dependency at all. `Plugins/` keeps the extension SPI: the interfaces, `PluginConfig`/`ConfigValue` (the payload a plugin author receives — `PluginConfig.Bind<T>()` binds it onto an options object IConfiguration-style: snake_case→PascalCase, dotted keys→nested objects, identifiers→enums, `required` enforced, all findings as `Result<T>` diagnostics), `ScaffoldContext`, and `PluginHandshake`. Config references Plugins (a configuration configures plugins), never the reverse. `PluginHandshake.Validate(assembly)` (`NSchema.Plugins`) is the engine handshake — the compile-time NSchema.Core reference is the metadata: majors must match, and the engine must be at least as new as the plugin's build target (an older plugin on a newer engine is fine; feature gaps surface as `Unsupported` renderings at plan time).

Diagnostics are minted through internal per-area `{Area}Diagnostics` catalogs (`ApplyDiagnostics`, `ProjectDiagnostics`, `TemplateDiagnostics`, …) living in the producer's folder — factory methods take the variable context as arguments **in its domain types** (`SqlIdentifier`, `Script`, `ChangeEvent`, … — never pre-unwrapped primitives; rendering a value object into display prose is the factory's job, so callers pass what they hold), pinning each area's source string and wording once. Policies and `DoctorOperation` are the deliberate exceptions (their diagnostics are the class's product; single-file cohesion wins). Tests assert cataloged diagnostics by record equality against the factory, not `Message.Contains`.

`DiffReader` is a **public, stateless utility, not a DI service** — Core never consumes it; it exists for consumers (the CLI's presenter is its only user). It's a static class (like `NsqlReader`/`NsqlWriter`/`NsqlFormatter`), so a caller reads a diff without touching the container. `Read(DatabaseDiff)` returns a `DiffDocument` — an ordered list of `DiffLine`s, each carrying its `ChangeKind` (null for a blank spacer), nesting `Depth`, and already-formatted `Text`, plus the aggregate `DiffSummary`. The document is **renderer-neutral**: a front-end folds it into colour, Markdown, or plain text by mapping each line's kind, never re-parsing rendered text. There is no `IDiffReader` interface and no `Use*` builder method. Core ships **no** schema or SQL-plan text renderer — those `SchemaRenderer`/`SqlPlanRenderer` dumps were trivial presentation with no reuse value and moved to the CLI; a consumer wanting to render those models (or a different diff format) writes its own.

### Operations

Each operation is its own vertical slice: an **internal** `IOperation<TArgs, TResult>` handler (`Execute(args, ct)`), a **public** `{Name}Arguments` record (the discoverable home for its inputs, empty where it has none yet), and a **public** `{Name}Result` record. Handlers are registered as `IOperation<{Name}Arguments, Result<{Name}Result>>` and invoked through the public `INSchemaOperations` facade (`app.Operations`) — not by user code resolving the handler. Operations return `Result`/`Result<T>` and never throw for expected outcomes, print, lock, or prompt: they narrate transient progress through `IProgress<OperationProgress>` and leave rendering, locking, and confirmation to the caller. There is no shared non-generic `IOperation` marker and no `OperationKind` enum — adding an operation means a new slice folder (handler + arguments + result), a `TryAddSingleton<IOperation<…>, …>()` registration, and a method on `INSchemaOperations` / `NSchemaOperations`.

Built-in operations (`src/NSchema.Core/Operations/`):

- **`PlanOperation`** (`Plan`) — computes a plan without executing. **A plan always diffs recorded state against the target; there is no planning against the live database.** `PlanArguments.Target` (`PlanTarget`) selects the state to converge *towards*: `Project` (the default — the declared schema; recorded state is empty at bootstrap, so a first plan previews a full create) or `Empty` (a teardown, the `terraform plan -destroy` analogue). A teardown is not a third kind of plan — it is the same path with nothing on the desired side, so it reads no project and obeys `Scope` like any other plan (a partial teardown is a scoped one); it converges **the managed set** towards nothing, never objects NSchema doesn't manage. Planning requires a configured state store (the diff is computed against the observation plus the run-once ledger and the managed set). **The plan's current side is the observation filtered to the managed and declared identities** (plus rename sources), computed in the planner; closure/widening still consults the full observation, so an unmanaged dependent that physically blocks a drop is still seen. The plan carries `MigrationPlan.Managed` — the managed identities an apply establishes: within the plan's scope, exactly what the project declares; outside it, whatever was already managed. To plan against current reality, `Refresh` first — the read-only live plan is deliberately gone, and the canonical CI recipe is plan-against-recorded plus a drift check. `OutFile` also writes a **saved plan file** (forward or teardown; a blocked plan is still written — apply enforces the policies again, so the file is a review artifact, not a bypass).
- **`ApplyOperation`** (`Apply`) — requires a configured state store (state records the run-once ledger, which is not rebuildable; a disposable database opts out explicitly via `UseEphemeralState()`), re-runs the plan policies against the carried plan (policy errors block unless `ApplyArguments.Force`, which demotes them to warnings), then executes `ApplyArguments.Plan`'s statements via `ISqlExecutor` and captures state — including the plan's managed set, empty plans included (that is how declaring an already-identical object adopts it); the plan comes from a plan operation or a saved plan file (applied verbatim, no recompute — the policy check is the only recomputation). **Applying against current reality is `Refresh` → `Plan` → `Apply`** (which refreshes again) — the leading refresh is the live read planning used to do implicitly, written down honestly, and a failed apply then leaves state reflecting pre-apply reality instead of stale. A saved teardown flows through this same path, so **tearing down is `Plan(Empty)` → `Apply`** — there is no separate destroy operation. A teardown diff is fully destructive, so the default destructive-action policy blocks it: applying one means setting that enforcement to `Allow` (which says exactly what is meant), not `Force` (which blanket-demotes every policy error, data hazards included).
- **`RefreshOperation`** — captures the live schema to the state store without planning or applying.
- **`ValidateOperation`** — loads the project and validates it against registered `IProjectPolicy` implementations; no planning/applying.
- **`DriftOperation`** — reads recorded (offline) state and live (online) schema and compares (direction recorded → live, so an out-of-band add reads as `Add`), returning the `DatabaseDiff`. Requires both a state store and a live provider. A pure observation — no policies run. The analogue of `terraform plan -refresh-only`.
- **`ImportOperation`** — fetches the live schema (optionally restricted by `Scope`) and writes it under `OutputDirectory` as SQL DDL via `DdlWriter`, **one file per major object** (e.g. `app/tables/users.sql`, `app/routines/add_tax.sql`), with schema-level objects in a per-schema header (`app.sql`) and database-global extensions in top-level `extensions.sql`. Additive: absent objects are left in place, re-imported objects merge (incoming wins). The file-split/merge logic lives directly in the handler — no `ISchemaImportTarget` seam.
- **`DoctorOperation`** — runs read-only health checks against the configured infrastructure and reports the outcome of each.

There is no `Show`, `Destroy`, `PlanDestroy`, or `ForceUnlock` operation. Previewing and applying a teardown are `Plan(Empty)` / `Apply` above; reading recorded state or a saved plan, inspecting or force-releasing the lock, and state surgery (pull/push, taint/untaint) are thin front-end reads and read-mutate-write loops over the public seams — `app.Database`, `app.PlanFile`, `app.Locks`, and `app.State` — not Core operations. Confirmation is likewise a caller concern (the CLI prompts); Core operations neither prompt nor lock.

**`IMigrationWorkflow`** (`Operations/Services/`) is the imperative shell around the pure planner shared by `Plan`, `Apply`, `Refresh`, and `Validate`: `ComputePlan(target, scope)` resolves the desired side (the project, or nothing for a teardown) and the recorded current side, then runs `IMigrationPlanner` (returning `Result<MigrationPlan>`), `Validate()` runs just the project policies, and `Refresh()` captures the live schema to the store, returning `Result<StateCapture>` (a failure when no store is configured). `Drift`, `Import`, and `Doctor` do **not** use the workflow — they talk to `IDatabaseProvider` / `IDatabaseComparer` / the importer directly. The workflow does not lock, confirm, or render.

### Schema providers

**The project** — the declared desired state — comes exclusively from SQL DDL files ("desired" needs no qualifier: the project *is* what you want). `AddProjectSource(baseDirectory, glob|Matcher)` registers a `DdlSchemaSource`; the **public** `IProjectProvider` (`app.Project`, default `ProjectProvider`) globs every registered source, reads matched files with `DdlReader`, and aggregates into a `ProjectDefinition` (public) — the declared `Database` **plus** the `ProjectDirectives` that steer its management. **Scripts are directives** (they steer a run, they don't declare state), so they live in the directives beside the renames/drops/partials: change-event scripts on `ProjectDirectives.ChangeScripts` (each addressing the table member it prepares), deployment (bookend) scripts on `ProjectDirectives.DeploymentScripts`. **The `SCRIPT` statement** (`SCRIPT <name> RUN [ALWAYS | ONCE] ON <event> AS $$…$$;` — the name is an identifier like any other object name) is the only script form. The script model is an abstract `Script` (name + SQL + scope + run semantics) with two concrete kinds — the kinds are never aggregated into one `Script` collection; each lives in its own typed home: `ON PRE|POST DEPLOYMENT` projects to a `DeploymentScript` (with `DeploymentPhase`), `ON ADD COLUMN|ALTER COLUMN TYPE|ADD CONSTRAINT <path>` to a `ChangeScript` (with `ChangeTrigger` + schema/table/member path — the "data migration" case). The event axis *is* the change-vs-deployment axis, so there is no separate `ScriptEvent` — the two script types carry what they run against directly. **The run-once ledger is a deployment-script concept.** `RunCondition {Always, Once}` lives on `DeploymentScript` alone; the `RUN [ALWAYS|ONCE]` clause is valid only before a deployment event and the AST node's constructor enforces that (a change event takes a bare `RUN`, `UNLESS EXISTS` is reserved). **A change-event script has no run condition and never enters the ledger:** its run is gated by the diff — it rides its change only when that change is planned, and after the change applies it is gone from the diff, so it re-runs exactly when the schema diverges enough to re-trigger it (which is correct). **`RUN ONCE` on a deployment script is ledger-enforced, and diffing is symmetric:** the differ compares what-I-have to what-I-want, where `CurrentState` (internal, `Diff/`) is the current schema **plus** the recorded executions — so the diff carries schema changes *and* script runs in one artifact. An already-executed run-once **deployment** declaration is not part of the current→desired difference, so the planner drops it (a changed body warns and stays skipped — rename to re-run); a change-event script is never dropped this way. The diff carries every script the difference implies exactly where it runs — each surviving change-event script inlined on its diff node (matched-only by construction; an unmatched one is a dead migration and not part of the difference), the deployment scripts on the root `DeploymentScripts` list. The state envelope carries the ledger as `scripts`, each entry recording the structural `script: {schema, name}` address (envelope stays version 1; payloads without it — including pre-5.0 `executedScripts` payloads — read as an empty ledger). `ScriptExecution` (script address + body hash + when) is **shared script vocabulary** (`Project/Domain/Models/Scripts/`) — an execution record is identity, not declaration (the body may since have changed or gone) — used by the ledger and the differ alike, so the recorded executions pass straight through to planning with no boundary translation. **The `MigrationPlan` is the complete executable artifact** — the complete diff (scripts included) and the statements travel as one, so everything that moves the plan (plan result, plan file, apply arguments) moves all of it; there is no separate manifest and no parallel run-once parameters anywhere. A successful apply passes the applied plan to `IMigrationWorkflow.Refresh(applied, force, ct)`, which records the run-once ledger through the state-domain method `DatabaseState.RecordExecution(Diff.DeploymentScripts, when)` (deriving one entry per run-once deployment script — address + hash — is the state model's job, so the shell only supplies what ran and when); a failed or empty apply passes nothing and records nothing. Refresh never rebuilds the ledger, and an unreadable existing payload **fails the capture unless forced** (`RefreshArguments.Force`; apply's post-execution capture always forces — the SQL has already run) — a forced replacement resets the ledger and rides the successful capture `Result` as warning diagnostics (the read errors demoted to warnings — error severity would flip the result to a failure), which the refresh/apply operations forward onto their own results. Planning requires a state store (the ledger is half of the current state). `Script` exposes a public computed `Hash` — the canonical body hash (SHA-256 of the UTF-8 text, computed by the model itself) that a run-once execution is recorded under; it is the **only** hash surface. The pre-4.4 spellings (`PRE|POST DEPLOYMENT '<name>' AS $$…$$;`, `MIGRATION ['name'] FOR <trigger> <path> AS $$…$$;`) were **removed in 5.0**: each now fails the parse with an error naming the `SCRIPT` replacement, and script names are required (anonymous migrations died with the old grammar). `GetProject` returns `Result<ProjectDefinition>` — the pure planner receives only the `ProjectDefinition` payload, and `MigrationWorkflow` merges the read diagnostics into plan/validate outcomes. Read failures ride that seam as error diagnostics — an unreadable or unparseable file (position and file name in the message), and no-files-matched — all reported at once, with the readable files' best-effort merge still carried; `DdlSyntaxException` is internal parser bail machinery and never crosses a public surface. **Script identity is `(scope, name)`** — `ScopedAddress` is the address (schema null = global; renders `schema.name` — unambiguous because names are identifiers), `Script.Address` computes it from the event's `ScopeSchema`, and the ledger lookups (`FindExecution`/`RemoveExecution` on `DatabaseState`) take it. Addresses are validated unique after template expansion; the same name in two scopes is two scripts. `GetProject(scope, ct)` is its one method — a public *read* of the resolved project (the CLI's `script untaint` takes declaration hashes from it); *supplying* desired schema from code remains deliberately unsupported — DDL files are the only input, and there is no `ISchemaTransformer`. Multiple `AddProjectSource` calls aggregate (e.g. a base set plus an environment overlay).

The language stack (all in `Project/Nsql/`): `NsqlLexer` → `NsqlParser` (`Project/Nsql/`, partials per statement family, pure syntax — emits the `Project.Nsql.Syntax` tree, template bodies unexpanded) → `DocumentProjector` (`ProjectStatement` projects one **declaration** into the domain through `DatabaseAccumulator`; directives — scripts among them — are the `DirectiveCollector`'s job, dispatched there once) → `ProjectAssembler` (`Project/`, a staged pipeline over **one `DatabaseAccumulator` spanning every document**: collect — routing each statement to the accumulator or the `DirectiveCollector`, validating template bodies — then expand the applications via `TemplateExpander` (a template instantiates by projecting its declaration nodes once per applied schema and feeding its directive nodes to a per-instance collector, unqualified names binding to that schema at projection; the instance's objects absorb into the same accumulator — a collision is the ordinary per-add duplicate, prefixed with its `APPLY TEMPLATE` — while its directives ride back into the one collector), then build once, resolve includes, and run the project-wide script validations. The file boundary is organization, not semantics: duplicates are caught per add with the position and file of the offending re-declaration; standalone triggers, indexes, and table grants attach project-wide, so they may live in a different file — or template — than the relation they target; and a second `CREATE SCHEMA` anywhere is a duplicate, since objects land in a schema from any file without redeclaring it). **`DirectiveCollector` is the single directive-dispatch point** (`TryAdd(statement, context?)` — the optional context is the applied schema a template instance binds unqualified addresses to); scripts and renames all land there, and `Build()` produces the whole `ProjectDirectives`. There is no per-document projected model: a single document is assembled the same way, so import and the test parser both go through `ProjectAssembler.Assemble` for a `ProjectDefinition`. The public read seam is **`NsqlReader`** (`Project/Nsql/`): `Read(source)` and the file-aware `ReadFile(path)` return `Result<NsqlDocument, NsqlDiagnostic>` — a `Result<T>` whose diagnostics are typed, each carrying its `SourcePosition` (and `File`, stamped by the reader) structurally; the parser recovers at statement boundaries, so every syntax error in a file reports at once, and a failed parse still carries the statements that parsed. **Schema templates** (`TEMPLATE name BEGIN … END;` + `APPLY TEMPLATE name IN SCHEMA a, b;`) declare a reusable object group instantiated per schema: bodies are fully parsed (no views/schemas/extensions inside; unqualified names bind to the target schema, qualified ones escape), and the internal `TemplateExpander` expands applications during project assembly after all sources aggregate — location- and order-agnostic, invisible to diff/plan/import. **The template vocabulary is fixed**: grammar words are nouns, never machinery verbs — an *application* (`APPLY TEMPLATE`), an *include* (`INCLUDE`); the engine *expands* an application (*instantiating* the template per applied schema and *absorbing* each instance into the accumulator/collector) and *resolves* an include. A template body holds the statements whose subject is an object *within* a schema — declarations, `SCRIPT`s, and the **object-level renames** (table/column/enum/domain/type/sequence/routine), each instantiated per applied schema; the schema-level and view renames are rejected in a body — a template describes a schema's contents, not the schema itself. `SCRIPT`s of either event kind are allowed — change-event scripts use an unqualified `table.member` path, and the table need not be declared in the same template (another template applied to the same schemas may declare it — matching is decided at diff time); deployment-bookend scripts instantiate one `DeploymentScript` per applied schema, scoped to it (a hand-written deployment script is global), so a scoped read filters every script by its event alone. The `{schema}` token substitutes in the SQL body only — instances keep the declared name, kept distinct by their scope. **Table templates** (`TEMPLATE name FOR TABLE BEGIN <members> END;`) hold comma-separated table members instead of statements and merge into a table via an `INCLUDE name` member (columns land at the include position); includes resolve after the schema-template applications expand (the internal `IncludeResolver`, a phase distinct from expansion), so a templated table can itself include one. Both template kinds share a `TemplateStatement` base (one name space, one registry). Duplicate index-backed names per schema (a hazard when a mixin carries an index) are caught by `StructuralIntegrityPolicy`, templates or not. `NsqlWriter` renders domain → syntax → text (`SyntaxBuilder` is its first half; Import constructs per-file documents through it). `NsqlFormatter` is a gentle token-stream reformatter sharing `NsqlLexer` (backs the CLI `fmt` command) — it pretty-prints without going through the full parse/model round-trip, so it preserves what it doesn't understand; statement-opener keywords are single-sourced in `NsqlKeywords`, shared with the parser's dispatch.

The **current side** of a diff has two sources, split into two feature clusters. The live database is **`Deployment`** — `IDatabaseProvider.GetLive(scope)` (`app.Database`), which reads the registered `IDatabaseIntrospector` (via `UseDatabaseIntrospector<T>()`), fails if none is configured, and re-applies the scope (the introspector's scope is an over-return-allowed hint). The recorded database is **`State`** — a plain `IDatabaseStateManager` read (the manager already carries the not-configured and unreadable-payload failures, so callers read state directly rather than through a wrapper; an absent payload is an empty database, so a first plan shows a full create). There is no fallback between the two, and **no axis choosing between them**: each caller reads the source it means (`ComputePlan` reads state once, serving both the recorded database and the ledger; `Drift` diffs a state read against a live read). Planning is a pure state read, so the DAG runs one way — **Deployment → (refresh) → State → (plan) → Diff** — and `Deployment` feeds only refresh, drift, and import.

**Scoping is a `PlanningScope`** (`NSchema.Model` — `All`, or `To(names)` with exact-name membership; an empty set normalizes to `All`). **A scope is an intensional *cover*, an `IdentitySet` an extensional *record*** — the same entry means different things: a schema in a scope covers everything inside it (`Covers(schema)` / `Covers(ObjectIdentity)` — membership implies downward through containment, and extensions, being database-global, are covered by every scope), while a schema in the managed set covers only the container (exactness is what replaced PARTIAL). They meet at one bridge (`IdentitySet.CoveredBy(scope)`) and one tree filter (`Database.ScopedTo(scope)` = `FilteredTo(Identities().CoveredBy(scope))`); growing the scope to object-granular members (targeting) only ever changes `Covers`. Scope applies symmetrically to every source read: `IProjectProvider.GetProject(scope)` (load-bearing — template instances bind at aggregation, so a project cannot be re-scoped later), `IDatabaseIntrospector.GetDatabase(scope)` (an optimization hint — an introspector may over-return, and `DatabaseProvider` re-applies the scope after every read, so scoping semantics stay single-sourced), and operation arguments (`PlanArguments.Scope`, `DriftArguments.Scope`, `ImportArguments.Scope`, default `All`). An unrestricted plan derives its scope in the workflow from `ProjectDefinition.AddressedSchemaNames` (the declared schemas plus every rename's current name) unioned with the schemas holding managed identities in state — a schema whose declarations were all removed stays under management until its drops apply. The project-side filter scopes directives too (an object directive stays in scope through either side of a schema rename).

Registering a state store via `UseStateStore*()` is all that's needed to enable offline planning. `UseEphemeralState()` registers an in-memory store (and matching lock) for disposable databases — integration tests, container bootstraps — which satisfies apply's state-store requirement without persisting anything beyond the application instance.

`IDatabaseStateStore` deals only in **serialized payloads** (`Read`/`Write` of `ReadOnlyMemory<byte>`) — the backend SPI, a persistence sink (file, blob, …) that never sees the schema model. Core owns the format behind the consumer-facing seam, **`IDatabaseStateManager`** (`app.State`, the **`State`** cluster root alongside its arguments/result records — the storage capability's consumer surface), which composes the store with the internal `IDatabaseStateSerializer` and deals in the **public `DatabaseState` model** (`State/Domain/Models/`, with locking in `State/Locks/` — the captured schema (the full observation, managed or not), the run-once execution ledger (`RecordExecution`/`RemoveExecution`/`FindExecution` mutators — a `RecordExecution` overload derives the ledger entries from a run's applied deployment scripts), and the managed `IdentitySet` (`Managed` — what an apply has created or adopted; with the ledger, the second piece of state that is history, not a rebuildable cache; a plain refresh preserves it, an applied plan replaces it with `MigrationPlan.Managed`)). The identity vocabulary is kernel model: `ObjectIdentity(ObjectKind, ObjectAddress)` and `IdentitySet` (schemas/objects/extensions with `Union`/`Except`/contains), with `Database.Identities()` and `Database.FilteredTo(IdentitySet)` as the model-owned behaviors. Its typed `Read`/`Write` are the primary surface — front-ends and library consumers do state surgery as pull → mutate → push (taint = remove a ledger entry, untaint = record one; adopt/forget = union/except on `Managed`); `ReadRaw`/`WriteRaw` move the payload uninterpreted for byte-shuffling (a raw push validates the payload deserializes, then writes it **verbatim** — never a re-serializing round-trip). Every member takes an `{X}Arguments` record and returns `Result<{X}Result>`: no store configured and an unreadable payload are failures, not exceptions — except `ReadRaw`, which never interprets, so corrupt state can still be pulled for repair. The seam does not lock (caller-managed, as with operations). `MigrationWorkflow` consumes the manager (a plan with an unreadable ledger fails rather than re-planning recorded scripts; a *forced* refresh is the corrupt-state recovery path — see the scripts passage above); `DoctorOperation` alone still probes store + serializer directly — distinguishing reachable-from-valid is its job. `DatabaseStateEnvelope` (the versioned wire format) stays internal.

### State locking

`IStateLock` (`State/Locks/Backends/IStateLock.cs`) is the **backend seam** coordinating exclusive access to shared state. It is separate from `IDatabaseStateStore`, but one backend may implement both. `Acquire(StateLockRequest, ct)` returns an `IStateLockHandle` and throws `StateLockedException` (carrying the holder's `StateLockInfo` when readable) when the lock is already held; `Release(ct)` drops whatever lock is held; `Peek(ct)` reads the holder without acquiring. The handle exposes `Info` and an idempotent `Release(ct)` — it is **not** `IAsyncDisposable`, so a caller can acquire a lock and never release it to hold it past the current process (a long-lived manual lock).

Front-ends never touch `IStateLock` directly; they go through `IStateLockManager` (`app.Locks`), which wraps it: `Acquire(LockAcquireArguments)` (operation name, optional TTL, and `SkipLock` in one record — `StateLockRequest` stays the backend currency and never carries the skip) returns a `Result<IStateLockHandle>` (a failure carrying the holder's details when already locked; a no-op handle when there is no backend, or on a skip-lock run — the latter carrying a warning naming the lock it ran past), and `Peek` / `Release` (force-release) mirror the backend. **Locking is caller-managed** — Core operations never lock; the CLI acquires the lock through `app.Locks` around the state-mutating runs it guards (apply, refresh, and destroy-via-apply) and releases it in a `finally`.

**The lock is registered automatically alongside the store.** `UseFileStateStore(path)` also registers a matching `FileStateLock` at `<path>.lock`; `UseStateStore<T>()` / `UseStateStore(instance)` register the same instance as the lock when the backend also implements `IStateLock`. An **explicit** lock choice (`UseStateLock*` / `UseFileStateLock`, tracked via a private `_explicitStateLock` flag) is never overridden by store registration, regardless of call order. With no store and no explicit lock, **no `IStateLock` is registered** and the manager's `Acquire` yields a no-op handle (there is no `NoOpStateLock`). (`FileStateLock` is a local-dev lock-file, not a distributed lock.)

### Planner pipeline

`MigrationPlanner` (`Plan/Domain/MigrationPlanner.cs`) conducts the pipeline, producing `Result<MigrationPlan>` — the complete artifact — with any `PolicyDiagnostic`s riding on the `Result`. **A policy block is a failure that still carries the complete plan** ("may not apply", not "stopped computing"): every stage runs regardless, and error severity is what blocks application — the CLI decides how to act. **The line between the stages: Diff answers "what is different" — schema changes and the script runs they imply; Plan answers "how it executes" — order and SQL.** Stages:

1. **Project stage** — runs every `IProjectPolicy` against the declared `ProjectDefinition` (schema plus scripts). Also exposed standalone as `IMigrationPlanner.Validate(desired)` (used by the validate operation).
2. **Diff stage** — `IProjectComparer` produces the complete `DatabaseDiff`: `IDatabaseComparer` emits the structural tree directly (no flat-action intermediate), run-once resolution drops already-executed **deployment** declarations (their skip warnings are diff-stage diagnostics; change-event scripts have no ledger and pass through, gated by the compare), and the matcher inlines each change-event script on the node it prepares while the deployment scripts ride the diff's root `DeploymentScripts` list (an unmatched change-event script is a dead-migration info).
3. **Linearize stage** — `IPlanLinearizer` (default `PlanLinearizer`) walks the diff and emits `MigrationAction`s in a safe dependency order, weaving the diff's own scripts in as `ExecuteScript` actions: each change node's inlined `MigrationScript` splices at that change, the root's deployment scripts bookend the list (pre first, post last).
4. **Render stage** — the planner walks the actions in order and renders each through the required `SqlDialect` (one action in, one or more statements out — decomposition, never reordering), scripts included: an `ExecuteScript` typically renders as its verbatim `Statement` (the action carries that default), but a dialect may validate or normalize the SQL. A rendering's diagnostics ride the plan result — an action the dialect doesn't support is an error that blocks application, not an exception. The artifact is the diff plus the statements; the actions themselves are discarded.
5. **Plan stage** — runs every `IPlanPolicy` against the complete artifact, post-render, so policies see exactly what an apply would execute (the diff, its scripts, and the statements) — a matched backfill legitimately suppresses the data-hazard warning because script coverage rides the plan's diff. The built-ins run here: `DestructiveActionPolicy` (enforcing `DestructiveActionOptions.Policy`), `DataHazardPolicy`, and `EnumValueRemovalPolicy`. Apply re-runs the same policies against the plan it is given.

There are **no transformer seams**: the project, diff, and plan are each validated by policies but never rewritten by the pipeline. The two policy seams sit on the two boundary artifacts — validate what you declared (`IProjectPolicy`), validate what you're about to run (`IPlanPolicy`); the diff needs no seam, it rides the plan.

## When writing documentation and XML/code comments.

- Public facing changes should be included in CHANGELOG.md.
- Don't overdocument everything. A brief, single-line summary is enough for XML docs. In some advanced cases where explanation is actually needed, then add another sentence or two in a Remarks block.
- When writing code comments, be aware that repeating the same reasoning in multiple places creates coupling and very quickly becomes out of date, so keep comments restricted to what actually matters at the place they're written.
- Don't expose implementation details in documentation or changelogs, as documentation is a form of contract, so changing an implementation becomes a breaking change.
- Stop confirming negatives. If I ask you to change something like "you've accidentally coupled these two services", don't then go and add five comments in various places saying "these two classes aren't coupled", because nobody was expecting them to be in the first place.
